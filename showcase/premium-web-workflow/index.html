01 / Reference Scout

先找样本

当用户没有给具体参考时，Codex 不应该立刻写 HTML。它应该先把业务内容翻译成搜索条件，从 Awwwards、Godly、MotionSites 等来源里找到可借鉴的结构。

这次参考

AW Portfolio

借它的动效作品集气质：Intro animation、Hero animation、3D gallery、Work gallery、滚动中的内容切换。

  • 不复制品牌、文字、图片和完整页面结构。
  • 只拆布局机制、空间关系、动画节奏和内容组织。
搜索输入

用户最多补一句

不要让用户填设计问卷。只要问清首屏气质：科技沉浸、苹果极简、奢华报告、游戏感、数据决策舱，或者由 Codex 自动先选。

输出形式

给出 3 个候选方向

每个候选都说明：为什么适合、能借什么机制、有什么风险。用户选定后再进入深拆和实现。

02 / Hero Gate

先做首屏

首屏不是一个大标题加两颗按钮。它要先成为一个能被截图判断的视觉场：有主体、有空间、有入口，也有下一屏的暗示。

规则 01

中文标题先设计

展示级标题不能让浏览器随意换行。先压缩成对仗短句，再检查桌面和移动端。

规则 02

图像不是贴图

图片负责氛围和主体；按钮、表格、图表和业务判断必须留在网页层。

规则 03

不要同一套模子

换颜色不等于换布局。参考必须改变结构、滚动、密度和节奏。

规则 04

不止首页漂亮

参考给的是骨架，内容区要沿用同一视觉系统继续做完。

03 / Production Flow

拆解再建

这套流程要把“审美”变成可执行步骤。不是说一句高级，而是让 Codex 每一步都有明确产物和失败判定。

01

参考侦察

把内容类型、气质、视觉主体、禁忌翻译成搜索条件。

02

机制深拆

拆首屏构图、滚动状态、图库结构、文字密度和交互节奏。

03

Hero 生产

默认使用生成图、自有图像、视频或代码场景，不用空洞背景冒充。

04

内容延展

参考结构要服务我们的内容：参考、规则、流程、验收、安装都要完整。

05

截图复盘

如果首屏像 PPT、标题乱、导航失效、移动端挤压，就回到设计层重做。

04 / Review System

验收不靠感觉

好看不能只靠一句“高级”。这个页面必须被浏览器截图检查：桌面、移动端、锚点跳转、标题换行和内容完整度都要看。

05 / Install

安装后默认使用

这套 Skill 的目标是让“做网页”默认进入参考驱动流程：先找参考、再定 Hero、再重建内容、最后截图验收。

不是网页模板库
